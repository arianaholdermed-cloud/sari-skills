# Feasibility Boundary Rules

## Core Rule
Revision plans must respect what is actually feasible before resubmission.

## Resource States
Always distinguish:
- currently feasible,
- potentially feasible,
- currently unrealistic.

## Important Rule
Potentially feasible is not the same as already committed.
Currently unrealistic requests should be handled with a bounded and transparent response strategy, not hidden.

## Reporting Rule
State clearly where the revision plan is resource-limited and where a bounded response is strategically safer than overcommitment.
